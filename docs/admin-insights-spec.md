# 后台「文件资产与访问洞察中心」规格 v1

> 状态：Phase 1 已完成并部署；Phase 3 仅在规模/治理需求出现后再评估
> 日期：2026-07-10  
> 适用范围：`/admin`、D1 `imginfo` / `tgimglog`、新上传链路

## 1. 要解决的问题

当前后台能筛选、预览、拉黑或删除资源，也能显示三组 Top 20；它更像一个“文件表格”。管理者仍很难回答：

1. 文件库有多大、主要由什么构成、占了多少可计量的容量？
2. 哪些文件正在被使用，访问是否在增长，主要来自哪里？
3. 一个具体文件的属性、热度、来源和治理状态分别是什么？
4. 哪些文件的数据不完整、长期无人访问、可能需要处理？

本迭代将后台定位为**文件资产与访问洞察中心**，不是通用 BI 工具。首页的任务是帮助管理者在 30 秒内决定下一步该看哪一批文件。

## 2. 成功标准与非目标

### 成功标准

- 管理者可在总览直接看到文件数、已采集容量、PV、独立访问 IP、活跃文件数与数据覆盖率。
- 可从热门资源、分布图和待关注项一键点入已筛选的资源中心或该文件详情。
- 每个文件都有可理解的详情面板：预览、基本元数据、访问摘要、来源与治理状态。
- “未采集”与“0”有严格区分；未知大小不能被误显示为 `0 B`。
- 新上传立即写入可由服务端可靠获得的元数据；旧文件通过可恢复的后台任务逐步补齐。

### 非目标

- 不把“质量”伪装成一个不透明的总分。
- 不在首期建立实时流处理、全量日志仓库或复杂自定义报表系统。
- 不因本功能改变 Telegram 的删除语义；TG 资源仍只会删除站内记录。
- 不为不存在的技术属性编造数据。尺寸、时长、编码等只在确实成功提取时显示。

## 3. 指标口径

所有统计均来自 D1 的受管资源和访问日志，而非运行时缓存。

| 指标 | 定义 | 首期呈现方式 |
|---|---|---|
| 文件总数 | `imginfo` 中受管资源总数 | KPI 卡片；可点入全部资源 |
| 已采集容量 | 有 `size_bytes` 的资源字节数之和 | 显示容量及“已覆盖 x / 总数 y”；覆盖不足时不显示为全库容量 |
| PV | 指定时间范围内 `tgimglog` 的访问记录数 | 7 天 / 30 天可切换，附与上个同长度周期的变化 |
| 独立访问 IP | 指定时间范围内不同 IP 数量 | 仅管理员可见；作为访问广度指标，不等同真实用户数 |
| 活跃文件 | 指定时间范围内至少有一次访问的不同 URL 数量 | KPI 与热门列表辅助指标 |
| 热门资源 | 先按指定范围 PV 降序；PV 相同则最近访问更近者优先 | 固定为可解释的排序，不首期引入黑盒热度分 |
| 增长最快 | 最近 7 天 PV 相对前 7 天的绝对增量 | 至少有 3 PV 才进入榜单，避免单次访问误导 |
| 长期未访问 | 已上传超过 30 天、且没有访问日志或最近访问早于 30 天 | 治理候选，不自动删除 |
| 数据完整度 | 每个类型要求字段的采集覆盖率 | 例如大小、规范 MIME、kind；以“缺什么”代替模糊质量分 |

### “质量”的三层解释

| 层级 | 问题 | 初期指标 |
|---|---|---|
| 技术属性 | 文件本身是否具备可用的技术信息？ | 大小、MIME、类型；成功时再显示尺寸/时长 |
| 数据完整度 | 管理数据是否足以安全管理？ | 必填字段覆盖率、类型不明、采集失败 |
| 使用价值 | 文件是否真的被使用？ | PV、独立 IP、访问增长、最后访问时间 |

这三层需要并列呈现，不能压缩成单一“质量分”。

## 4. 信息架构与页面行为

侧栏沿用当前“图床 · 运维台”的结构，只调整优先级：

1. **概览**：默认入口，新的资产与洞察总览。
2. **资源**：现有资源列表/网格的增强版，承担管理动作。
3. **访问日志**：保留原始访问排查能力，作为详情的下钻入口。
4. **API**：保持现有密钥管理与文档。

### 4.1 概览

页面顶部为“文件资产概览”，带 7 天 / 30 天时间范围切换。范围只影响访问类指标，不影响文件总数与已采集容量。

首屏按从“全局判断”到“下一步动作”的顺序排列：

1. **KPI 行**：文件总数、已采集容量、PV、独立访问 IP、活跃文件。
2. **访问趋势**：按天 PV 折线/柱状图，支持悬停查看当天值；数据不足时显示说明而非空坐标轴。
3. **资源构成**：按文件类型与存储位置的数量分布；容量数据只在覆盖充分时补充。
4. **热门资源**：缩略图、文件名/路径、PV、相对上期变化、最后访问时间；点击进入详情。
5. **需要关注**：类型不明、缺少大小、长期未访问、采集失败。每项显示数量、原因和“查看资源”入口。
6. **数据覆盖提示**：明确告知“历史文件元数据正在补全”，避免用户把局部容量理解成总容量。

### 4.2 资源中心

保留表格/网格切换，但升级为有任务导向的资源浏览器：

- 搜索、来源、类型、状态筛选同步到 URL，刷新和分享链接后状态不丢失。
- 新增排序：最近上传、最多访问、最近访问、最大文件、最久未访问。
- 列表增加可选列：大小、最后访问、最近 30 天 PV、元数据状态。
- “热门资源”“待关注”点击后直接带筛选和排序进入资源中心。
- 批量删除、拉黑不属于本规格首期必做项；先确保每个单资源动作的后果清晰且可恢复策略明确。

### 4.3 文件详情抽屉

桌面端从右侧打开宽详情抽屉；移动端改为全屏页面态，避免在窄屏内塞入表格。

抽屉包含四个明确区块：

1. **资源头部**：预览、文件名/路径、类型、状态与复制链接。
2. **文件属性**：存储位置、上传时间、大小、MIME、尺寸/时长（若已采集）、元数据采集状态。
3. **访问表现**：7/30 天 PV、独立 IP、最后访问、迷你趋势图、主要来源。
4. **管理动作**：打开原文件、复制链接、查看完整访问日志、拉黑/解除拉黑、删除；危险动作沿用明确的二次确认和存储影响提示。

### 4.4 状态设计

- 加载：KPI 与列表使用骨架屏，保留页面结构，不整页闪烁。
- 无数据：说明“尚无访问记录”或“尚未采集大小”，并给出下一步，而不是只显示 `0`。
- 部分数据：以“已覆盖 16 / 140 个文件”这类文字披露覆盖范围。
- 失败：在对应卡片或区块内重试，不因一个聚合查询失败让整个后台不可用。

## 5. 数据模型与采集策略

### 5.1 保持现有表的职责

- `imginfo`：一条受管文件资产的主记录，继续保存 URL、上传来源、MIME、kind、状态、累计访问等。
- `tgimglog`：原始访问日志，继续用于排障、来源分析和初期统计。
- 运行时缓存不作为任何统计或资产字段的来源。

### 5.2 首期扩展 `imginfo`

| 字段 | 类型 | 说明 |
|---|---|---|
| `size_bytes` | INTEGER nullable | 文件字节数；未知为 `NULL`，不是 0 |
| `width` / `height` | INTEGER nullable | 仅图像或视频且成功提取时记录 |
| `duration_seconds` | REAL nullable | 仅音视频且成功提取时记录 |
| `sha256` | TEXT nullable | 后续去重候选；首期不要求历史全量计算 |
| `metadata_status` | TEXT | `complete`、`partial`、`pending`、`unavailable`、`failed` |
| `metadata_checked_at` | TEXT nullable | 最近一次采集/补全时间，ISO 8601 |

迁移必须保持现有资源可读。旧记录先标记为 `pending` 或 `unavailable`，不要在迁移中同步扫描所有 R2/TG 对象。

### 5.3 新上传与历史补全

- **新上传**：从上传请求直接写入服务端可可靠获得的文件大小、规范 MIME、kind 与采集状态；这是容量指标逐步可信的基础。
- **图片尺寸**：只在有可靠提取能力时写入。浏览器预读可用于体验优化，但不能把客户端传值作为唯一可信来源。
- **历史 R2 文件**：按小批次、可重试、可暂停任务读取对象头信息，优先补 `size_bytes` 和 MIME；每批记录成功/失败状态。
- **历史 TG 文件**：若无法以安全且低成本方式获得原始大小，标记 `unavailable`，在 UI 诚实展示，不进行昂贵的全量下载。
- **哈希与重复检测**：只对新上传或管理员显式触发的资源计算；不首期批量扫描。

### 5.4 访问统计的演进

首期可直接基于 `tgimglog` 按受控时间范围聚合，因为当前数据规模小。所有查询必须限定日期、`LIMIT` 和现有索引友好字段。

当日志规模使原始聚合变慢时，再增加按日聚合表：

```text
media_daily_stats(date, url, pv, unique_ip_count, last_accessed_at)
PRIMARY KEY(date, url)
```

原始日志仍保留短期排障所需窗口；日聚合表用于趋势与概览。不要在首期为了未来规模过早引入单独的实时分析系统。

## 6. API、性能与安全边界

- 用一个管理员保护的概览端点返回某时间范围的 KPI、趋势、分布、热门资源和治理计数，避免前端为一个页面发出多条耦合请求。
- 时间范围只接受白名单枚举（首期 `7d`、`30d`），所有 SQL 使用参数化绑定。
- 资源列表沿用分页；统计端点限制热门列表与趋势点数。
- 原始 IP 仅在管理员会话下展示。后续定义日志保留期限和匿名化策略前，不向普通用户或开放 API 暴露。
- 不将 Telegram/R2 的凭据、源文件内容或未授权直链写入统计响应。

## 7. 交付分期

### Phase 1：可用的洞察闭环（优先实现）

- 新概览页：KPI、30 天趋势、类型/存储分布、热门资源、待关注项。
- 资源 URL 状态同步、排序、从统计点穿。
- 文件详情抽屉：预览、既有元数据、PV/最后访问/来源和管理动作。
- 明确的空状态、部分数据状态与移动端详情体验。

### Phase 2：数据可信度

- D1 migration 与新上传 `size_bytes`、采集状态。
- R2 历史资源渐进式补全与覆盖率呈现。
- 详情中逐项显示已获得的尺寸/时长/Hash，不可得时说明原因。

### Phase 3：治理与规模化（需求验证后）

- 按日统计聚合、访问异常提示、来源维度趋势。
- Hash 重复候选、批量治理、可配置保留策略。
- 更深的类型质量规则与导出能力。

## 8. 验收清单

- [ ] 7 天和 30 天的 PV、独立 IP、活跃文件数使用一致且可测试的时间边界。
- [ ] 已采集容量永远显示覆盖范围；没有大小的文件不计作 `0 B`。
- [ ] 热门、增长最快和长期未访问的定义在页面帮助文案与实现中一致。
- [ ] 点击任何榜单或治理项可到达已带筛选条件的资源视图或详情抽屉。
- [ ] 文件详情在桌面与移动端可完成预览、判断状态和管理操作。
- [ ] 所有统计与详情接口保持管理员权限、参数化 SQL、受限范围和空/失败状态。
- [ ] 新上传不依赖客户端宣称的数据来决定服务端可信的容量与类型。

## 9. 实施决策记录

1. 首期不做“质量总分”，改用技术属性、数据完整度和使用价值三组可解释指标。
2. 首期不急于引入流式分析或日聚合表；以当前 D1 数据规模先完成正确的查询与 UI 闭环。
3. 容量和尺寸从“新上传即采集、历史渐进补全”开始；不以一次性全量扫描阻塞上线。
4. 先做好单资源详情和点穿，再评估批量治理，避免高风险操作早于信息可见性。
