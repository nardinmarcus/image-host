# 上传信任与媒体分发性能设计

## 目标

在保留 Namoo Pix 现有 teal Soft Studio 视觉语言的前提下，完成已审计的 P0/P1：用户在上传前知道文件去向和删除语义，上传时能理解每个文件的状态并重试，后台媒体读取不再信任 Referer 或缓冲整个 TG 文件。

## 前台体验

未登录状态以“登录后上传”作为唯一主动作，并把支持的类型与存储规则作为说明，而不是展示不可用的基础设施选择器。登录后，用户选择“可删除托管（R2）”或“发送到 Telegram 频道”；TG 模式在首次使用前必须确认其外部发送和删除边界。

选择文件后，每个文件都拥有独立状态：准备上传、上传中、成功、失败。失败文件可重试；上传完成后，结果区优先提供直链和 Markdown 的单项复制与全部复制。拖拽区本身可点击、可键盘操作；缩略图、复制与开关均提供清晰的语义控件。

## 分发与数据边界

被拦截资源只允许真实管理员会话预览，不能用 Referer 判断权限。R2 与 TG 资源均先以 D1 元数据确认资源仍存在，再读取本地缓存，确保缓存不是删除后的事实来源。访问日志/PV 写入交给 `ctx.waitUntil`，但拉黑判定保留同步。

TG 文件使用已存 MIME 推断内容类型并直接返回上游 `ReadableStream`，不再把整份文件读入 ArrayBuffer。运行时不再执行 `ALTER TABLE` 或逐条历史回填；索引由受版本控制的 D1 migration 创建。

## 验证

1. 无登录、R2、TG、失败重试以及桌面/手机断点均可完成核心流程。
2. 含伪造 Referer 的未认证请求仍无法读取拉黑资源；管理员会话可以预览。
3. D1 migration 可在本地数据文件上幂等应用；运行时源码不含 schema repair DDL。
4. TG 路由不调用 `arrayBuffer()`，并通过 Response stream 返回上游响应体。
5. `npm run lint`、针对关键状态/路由的测试以及浏览器审查通过。
